# lcrc (locoroco) — Local agentic-LLM benchmarking framework

## Context

There is no existing tool that lets a user **autonomously sweep local agentic LLMs across model × backend × harness × quantization × KV cache × context length**, on their own hardware, and emerge with a defensible recommendation per task. Existing solutions cover slices:

- **Inspect AI** (UK AISI) — agentic eval framework, has Ollama/llama-cpp/vLLM providers, 200+ pre-built evals, but no sweep/perf orchestration.
- **Bench360** — server-class backends (vLLM/SGLang/TGI/LMDeploy) with system metrics; no Ollama/LM Studio, no agentic harness.
- **mini-swe-agent** — minimal bash harness used as a fair baseline by Vals AI; not a framework.

The novel contribution of lcrc is the **autonomous orchestration layer** that ties these together, plus the **harness comparison axis** (which nobody covers publicly). The framework is local-first, runs on the user's hardware, and produces a Pareto-optimal recommendation report so the user stops guessing which model/backend/quant to run.

Intended outcome: a user runs `lcrc discover` once on their machine, then `lcrc run --profile=weekly` periodically as new models drop, and gets a markdown/HTML report telling them the best combination for each task type they care about.

## Design decisions (resolved)

| # | Decision | Choice |
|---|---|---|
| 1 | v1 harnesses | **OpenCode (subprocess) + built-in mini-swe-agent-style bash harness**; `Harness` protocol for v2 (Claude Code, pi.dev) |
| 2 | Stack | **Python everywhere**, strict typing, FastAPI + bundled SPA (single deployable) |
| 3 | Task model | **Two tracks**: `CapabilityTask` (Inspect-derived) + `PerfTask` (Bench360-derived). Joined in the report. |
| 4 | Sweep strategy | **Tiered (smoke → narrow → confirm)** with rule-based dropout. Axis pinning via profile to skip low-churn axes (harness/backend) on weekly runs. |
| 5 | Backend lifecycle | **Hybrid**: lcrc owns process lifecycle for headless backends (llama.cpp, Ollama via llama-swap). Build versions captured in run metadata. Pre-flight 1-token verification per cell. |
| 6 | Metrics | **Macro + power/thermal**. Tier-1 platforms: macOS Apple Silicon, Linux NVIDIA. |
| 7 | Storage | **SQLite + JSON blobs**. Inspect `.eval` files referenced by path. Machine fingerprint is a first-class entity. |
| 8 | Frontend | **Dashboard + run launcher + live progress**. React + Vite + TanStack Query + Tailwind + shadcn/ui. WebSocket for live, REST for everything else. |
| 9 | Recommendation | **Pareto-front per task** + 2–3 baked use-case profiles (coding, long-context, low-resource). MD / HTML / JSON outputs. |
| 10 | Model catalog | **Curated baseline + opt-in discovery** from Ollama library / HuggingFace. Hardware-fit gate skips cells that won't fit. |
| 11 | v1 backends | **Ollama (via llama-swap) + llama.cpp (via llama-server) only**. vLLM, LM Studio = v2 plugins. |

## The "sweep" concept

A **sweep** is one execution of the orchestrator. It takes a `SweepConfig` (a set of axis values) and produces a set of **cells**, where each cell is one concrete combination of `(model, backend, harness, params)`. The orchestrator runs each cell against the configured tasks, captures metrics + scores + transcripts, and writes everything to SQLite under a single `sweep_id`. A sweep is the unit of reproducibility, resumability, and reporting.

```
SweepConfig (axes)
   │
   ▼
Cell expansion  ──►  [cell_1, cell_2, ..., cell_N]   each = (model, backend, harness, params)
   │
   ▼
Per-cell loop:
   preflight (mem fit, build present, 1-token test)
   ├─ for trial in 1..N:
   │     start backend → run harness on tasks → collect metrics → grade → write to SQLite
   └─ teardown backend
   │
   ▼
Reporter   ──►   Pareto front per task + baked profile picks  (MD / HTML / JSON)
```

A sweep is described by a single configuration concept: a **profile** (TOML file). One file format covers every use case from "figure it out for me" to "run exactly this Cartesian product". Profiles are resolved either by **name** (looked up in `~/.lcrc/profiles/<name>.toml`) or by **path** (`--profile ./my-experiment.toml`), which covers both persistent personal preferences and ephemeral version-controlled experiments.

### Profile schema

Each axis in a profile can be in one of three states:

| State | Meaning | Example |
|---|---|---|
| `pinned: <value>` | Fixed; not swept. | `harness = { pinned = "mini_bash" }` |
| `from_catalog: true` | Orchestrator pulls candidate values from the curated catalog at runtime (filtered by the hardware-fit gate). | `model = { from_catalog = true }` |
| `values: [a, b, c]` | Explicit sweep dimension. | `quant = { values = ["q4_k_m", "q5_k_m", "q8_0"] }` |

Two top-level fields control execution policy:

- `dropout = "tiered"` (default) — Stage 1 smoke screen → Stage 2 quant/context expansion on survivors → Stage 3 top-K with N=3 trials on the full task suite. Cells crash, score below threshold, or get dominated on the Pareto front are dropped between stages with logged reasons.
- `dropout = "off"` — no dropout; runs the literal Cartesian product. Still honors budget/timeout/preflight.

### Shipped profiles

| Name | What it does | Equivalent of |
|---|---|---|
| `discover` | All axes `from_catalog`, `dropout = "tiered"`. | First-time autonomous discovery on a new machine. |
| `weekly` (template) | User pins backend + harness after first `discover`; model/quant/context stay `from_catalog`. | Recurring monthly runs as new models drop. |
| `smoke` | Pinned single combination, single tiny task, `dropout = "off"`. | CI smoke / integration test fixture. |

### Commands

| Command | What it does |
|---|---|
| `lcrc run --profile <name\|path>` | Canonical entry point. Runs the profile through the engine. |
| `lcrc discover` | Convenience alias for `lcrc run --profile discover`. |
| `lcrc profile init <name>` | Scaffolds a profile from a template (e.g., from the latest `discover` run's recommended top picks → produces a `weekly` profile with backend/harness pinned). |
| `lcrc profile list` / `lcrc profile show <name>` | Inspect saved profiles. |

All runs share the same engine, the same SQLite output, the same resume semantics, and the same reporter (which queries by `sweep_id` and is profile-agnostic).

## Detailed design

### Core data model

All data crossing module boundaries is a `pydantic.BaseModel` with `model_config = ConfigDict(frozen=True, extra="forbid", strict=True)` unless noted. Identifiers are `NewType`-wrapped `str`s so they don't get accidentally mixed.

```python
# src/lcrc/types/ids.py
SweepId    = NewType("SweepId", str)      # e.g. "swp_2026-04-29_8a3f"
CellId     = NewType("CellId", str)       # e.g. "cell_swp_..._042"
TrialId    = NewType("TrialId", str)
MachineId  = NewType("MachineId", str)    # sha256(stable hardware fingerprint)[:16]
TaskId     = NewType("TaskId", str)       # "humaneval", "mmlu_pro_slice", "perf.single_stream"
CatalogEntryId = NewType("CatalogEntryId", str)
BackendName    = Literal["llamacpp", "ollama"]   # extends in v2
HarnessName    = Literal["mini_bash", "opencode"]  # extends in v2
QuantTag       = Literal["q4_k_m", "q5_k_m", "q6_k", "q8_0", "fp16", "awq_int4", "gptq_int4"]
KVCacheQuant   = Literal["f16", "q8_0", "q4_0"]
```

### Core abstractions (Protocols)

Every plugin point is a `Protocol` so backends/harnesses/tasks/metrics-collectors can be swapped without touching the orchestrator.

```python
# src/lcrc/backends/base.py
class Backend(Protocol):
    name: BackendName

    def detect_build_version(self) -> str:
        """Return the build identifier of the installed binary (e.g. 'llama.cpp b3214')."""

    def prepare(self, spec: BackendSpec, model: ModelSpec) -> BackendHandle:
        """Spawn / configure the backend process for this exact spec.
        Returns once `/health` is OK and a 1-token preflight succeeded.
        Raises BackendStartupError on any failure (caller marks cell preflight_failed)."""

    def model_endpoint(self, handle: BackendHandle) -> ModelEndpoint:
        """Adapt the handle into a harness-friendly endpoint."""

    def teardown(self, handle: BackendHandle) -> None:
        """Best-effort cleanup; never raises."""

# src/lcrc/harnesses/base.py
class Harness(Protocol):
    name: HarnessName

    def detect_version(self) -> str:
        """Return the harness binary version (or our internal version for built-ins)."""

    def run(
        self,
        endpoint: ModelEndpoint,
        task: Task,
        workdir: Path,
        timeout_seconds: int,
    ) -> HarnessResult:
        """Execute the harness against the task. Writes transcript to workdir/transcript.jsonl.
        Raises HarnessTimeoutError on timeout, HarnessCrashError on internal failure."""

# src/lcrc/tasks/base.py
class Task(Protocol):
    id: TaskId
    kind: Literal["capability", "perf"]

    def manifest(self) -> TaskManifest:
        """Static info: name, description, expected runtime, dataset size."""

class CapabilityTask(Task, Protocol):
    def evaluate(
        self,
        harness: Harness,
        endpoint: ModelEndpoint,
        workdir: Path,
        n_problems: int | None = None,        # None = full dataset
    ) -> CapabilityScore: ...

class PerfTask(Task, Protocol):
    def measure(
        self,
        endpoint: ModelEndpoint,
        workdir: Path,
        duration_seconds: int,
    ) -> PerfMetrics: ...                     # PerfTasks never use a harness

# src/lcrc/metrics/base.py
class MetricsCollector(Protocol):
    def supported(self) -> bool:
        """Is this collector usable on the current host?"""

    def sample_window(self, pid: int, duration_seconds: int) -> list[MetricSample]:
        """Block for `duration_seconds`, return ordered time series."""

    def start(self, pid: int) -> CollectorHandle:
        """Begin background sampling. Returns a handle for later stop()."""

    def stop(self, handle: CollectorHandle) -> tuple[list[MetricSample], MetricAggregate]: ...

# src/lcrc/reporting/base.py
class Recommender(Protocol):
    def recommend(
        self,
        sweep_id: SweepId,
        per_task_results: dict[TaskId, list[TrialResult]],
        weights: ProfileWeights | None = None,
    ) -> list[Recommendation]: ...
```

### End-to-end data flow

What happens when a user runs `lcrc run --profile discover`:

```
1. CLI (cli.py)
   parses argv → Typer command `run` → loads profile by name from
   ~/.lcrc/profiles/discover.toml → builds RunRequest(profile, machine_fingerprint, db_path)

2. Orchestrator.start(RunRequest) → returns SweepId
   - Resolve profile (validate against schema, expand defaults)
   - Detect HardwareFingerprint, upsert into machines table
   - Compute candidate axes:
     · model: from_catalog → query CatalogService.list(filter=fit_gate(hardware))
     · backend: from_catalog → list of detected installed backends
     · harness: from_catalog → list of available harnesses
     · params: from_catalog → defaults per (model, backend)
   - Cell expansion (cartesian, deduped)
   - Insert sweep + cells rows in SQLite (status='pending')
   - Emit SweepStarted event

3. Orchestrator.run_loop(sweep_id)  (per-cell, per-stage)
   For each cell in stage:
     a. preflight:
        - hardware-fit check (model_size <= available_memory * 0.85)
        - backend.prepare(spec, model) → BackendHandle  (writes build_version into cell row)
        - if anything fails → mark cell preflight_failed, emit CellPreflightFailed, continue
     b. for trial_idx in 1..N:
        - emit TrialStarted
        - metrics_collector.start(handle.pid)
        - for task in cell.tasks:
            if task.kind == 'capability':
              task.evaluate(harness, endpoint, workdir) → CapabilityScore
              harness.run(...) writes transcript.jsonl, returns HarnessResult
            else:
              task.measure(endpoint, workdir, duration) → PerfMetrics
        - metrics_collector.stop() → samples + aggregate
        - persist TrialResult + metric_samples + metric_aggregate
        - emit TrialCompleted
     c. backend.teardown(handle)
     d. emit CellCompleted

   Between stages (tiered dropout policy only):
     - apply dropout rules (crashed | score < threshold | dominated)
     - log dropout reasons into stage_decisions table
     - emit StageCompleted with surviving cell ids

4. Reporter (sync, after sweep status='completed')
   - Recommender.recommend(sweep_id) → list[Recommendation]
   - render(format=md|html|json) → file or in-memory
   - emit SweepCompleted with report path
```

Concurrency rule: **only one cell runs at a time** in v1. Backends are heavy; serialization is the only safe default. Tasks within a cell run sequentially (Inspect's own concurrency for capability evals stays internal). Frontend WebSocket delivers events in order.

### Profile TOML format

A profile is a TOML file. Below is the canonical full schema, then the `discover` example.

```toml
# ~/.lcrc/profiles/discover.toml
[meta]
name = "discover"
description = "Full autonomous discovery sweep across the curated catalog."
version = 1                            # schema version

[execution]
dropout = "tiered"                     # "tiered" | "off"
trials_per_cell = 3
per_cell_timeout_seconds = 300
per_run_budget_seconds = 28800         # 8h
resume = true                          # auto-resume from last incomplete cell on rerun

[execution.tiered]
stage1_n_problems = 5                  # smoke screen size
stage1_quality_floor = 0.20            # drop cells scoring below this
stage2_pareto_keep = 0.50              # keep top 50% on Pareto front
stage3_top_k = 10                      # final round size

[axes.model]
from_catalog = true
filter.max_size_b = 14.0               # optional fit override
filter.families = ["qwen3", "llama3", "gemma3", "phi4", "mistral"]   # optional whitelist

[axes.backend]
from_catalog = true                    # use any installed backend

[axes.harness]
from_catalog = true                    # use any available harness

[axes.quant]
from_catalog = true                    # use catalog-supported quants per model

[axes.context_length]
values = [4096, 16384, 32768]          # explicit sweep

[axes.kv_cache_quant]
pinned = "f16"                         # default; rarely worth sweeping in v1

[tasks]
capability = ["humaneval_mini", "mmlu_pro_slice", "swebench_verified_lite"]
perf       = ["single_stream", "concurrent_burst"]
```

A pinned-axes example (typical `weekly` profile):


### Live-progress event schema

WebSocket messages are a single discriminated union, JSON-serialized. The frontend's `useRunStream` hook is typed against this schema. All events carry `sweep_id` and `event_id` (monotonic int per sweep, used for replay/reconnect).

```python
class ProgressEvent(BaseModel):
    event_id: int
    sweep_id: SweepId
    event_kind: Literal[
        "sweep_started", "stage_started", "stage_completed",
        "cell_started", "cell_preflight_failed", "cell_completed", "cell_dropped",
        "trial_started", "trial_completed",
        "metric_tick",
        "sweep_completed", "sweep_failed", "sweep_cancelled",
    ]
    timestamp: datetime
    payload: dict[str, Any]            # schema per event_kind, validated separately

# example payloads
class CellStartedPayload(BaseModel):
    cell_id: CellId
    model: str
    backend: BackendName
    harness: HarnessName
    eta_seconds: int
    queue_position: int
    queue_total: int

class MetricTickPayload(BaseModel):
    cell_id: CellId
    trial_id: TrialId
    t_offset_ms: int
    tokens_per_second: float | None
    cpu_pct: float
    ram_bytes: int
    vram_bytes: int | None
    power_w: float | None
```

Server emits events on a per-sweep `asyncio.Queue`; WebSocket subscribers receive a backlog from `event_id=last_seen+1` on reconnect.

### Storage schema (SQLite, full)

Schema is migration-managed (one file per version: `migrations/0001_init.sql`, etc.). Every migration ships with a forward-only test. SQLite pragmas: `journal_mode=WAL`, `synchronous=NORMAL`, `foreign_keys=ON`, `busy_timeout=5000`.

```sql
CREATE TABLE schema_version (version INTEGER PRIMARY KEY, applied_at TEXT NOT NULL);

CREATE TABLE machines (
    id            TEXT PRIMARY KEY,           -- MachineId
    fingerprint   TEXT NOT NULL,              -- HardwareFingerprint as JSON
    hostname      TEXT NOT NULL,
    first_seen_at TEXT NOT NULL,
    last_seen_at  TEXT NOT NULL
);

CREATE TABLE catalog_entries (
    id        TEXT PRIMARY KEY,               -- CatalogEntryId
    family    TEXT NOT NULL,
    size_b    REAL NOT NULL,
    description TEXT,
    payload   TEXT NOT NULL                   -- full CatalogEntry as JSON
);

CREATE TABLE profiles (
    name       TEXT NOT NULL,
    machine_id TEXT,                          -- NULL for shipped/global
    payload    TEXT NOT NULL,                 -- profile TOML as JSON
    updated_at TEXT NOT NULL,
    PRIMARY KEY (name, machine_id),
    FOREIGN KEY (machine_id) REFERENCES machines(id)
);

CREATE TABLE sweeps (
    id            TEXT PRIMARY KEY,           -- SweepId
    machine_id    TEXT NOT NULL,
    profile_name  TEXT,
    profile_snapshot TEXT NOT NULL,           -- full resolved profile JSON
    started_at    TEXT NOT NULL,
    completed_at  TEXT,
    status        TEXT NOT NULL,              -- pending|running|completed|failed|cancelled
    FOREIGN KEY (machine_id) REFERENCES machines(id)
);

CREATE TABLE cells (
    id              TEXT PRIMARY KEY,         -- CellId
    sweep_id        TEXT NOT NULL,
    stage           INTEGER NOT NULL,         -- 1, 2, 3 (or 0 for dropout='off')
    model_id        TEXT NOT NULL,
    backend         TEXT NOT NULL,
    backend_build   TEXT,                     -- captured at preflight
    harness         TEXT NOT NULL,
    harness_version TEXT,
    spec            TEXT NOT NULL,            -- full CellSpec JSON
    status          TEXT NOT NULL,            -- pending|preflight_failed|running|completed|dropped|errored
    dropout_reason  TEXT,                     -- when status='dropped'

    completed_at    TEXT,

    FOREIGN KEY (model_id) REFERENCES catalog_entries(id)
);

    id          TEXT PRIMARY KEY,             -- TrialId
    cell_id     TEXT NOT NULL,
    trial_idx   INTEGER NOT NULL,
    started_at  TEXT NOT NULL,
    ended_at    TEXT,

CREATE TABLE task_results (
    id              INTEGER PRIMARY KEY,
    trial_id        TEXT NOT NULL,
    task_id         TEXT NOT NULL,
    kind            TEXT NOT NULL,            -- capability|perf
    score_payload   TEXT NOT NULL,            -- CapabilityScore | PerfMetrics JSON

    grader_path     TEXT,                     -- Inspect .eval reference
    FOREIGN KEY (trial_id) REFERENCES trials(id)
);
CREATE INDEX idx_task_results_task ON task_results(task_id);

CREATE TABLE metric_samples (
    trial_id    TEXT NOT NULL,
    t_offset_ms INTEGER NOT NULL,
    cpu_pct     REAL NOT NULL,
    ram_bytes   INTEGER NOT NULL,
    vram_bytes  INTEGER,
    gpu_pct     REAL,
    power_w     REAL,
    PRIMARY KEY (trial_id, t_offset_ms),
    FOREIGN KEY (trial_id) REFERENCES trials(id)
);

CREATE TABLE metric_aggregates (
    trial_id        TEXT PRIMARY KEY,
    peak_ram_bytes  INTEGER NOT NULL,
    peak_vram_bytes INTEGER,
    avg_cpu_pct     REAL NOT NULL,
    avg_gpu_pct     REAL,
    energy_joules   REAL,
    ttft_ms_p50     REAL,
    ttft_ms_p95     REAL,
    itl_ms_p50      REAL,
    itl_ms_p95      REAL,
    itl_ms_p99      REAL,
    tokens_per_second REAL,
    sample_count    INTEGER NOT NULL,
    FOREIGN KEY (trial_id) REFERENCES trials(id)
);

CREATE TABLE stage_decisions (
    sweep_id    TEXT NOT NULL,
    cell_id     TEXT NOT NULL,
    stage       INTEGER NOT NULL,
    decision    TEXT NOT NULL,                -- promoted|dropped
    reason      TEXT NOT NULL,
    decided_at  TEXT NOT NULL,
    PRIMARY KEY (sweep_id, cell_id, stage),
    FOREIGN KEY (sweep_id) REFERENCES sweeps(id),
    FOREIGN KEY (cell_id) REFERENCES cells(id)
);

CREATE TABLE events (
    sweep_id    TEXT NOT NULL,
    event_id    INTEGER NOT NULL,
    event_kind  TEXT NOT NULL,
    timestamp   TEXT NOT NULL,
    payload     TEXT NOT NULL,                -- JSON
    PRIMARY KEY (sweep_id, event_id),
    FOREIGN KEY (sweep_id) REFERENCES sweeps(id)
);
CREATE INDEX idx_events_sweep ON events(sweep_id, event_id);
```

### Cross-cutting concerns

**Logging.** `structlog` with JSON renderer in production, console renderer in dev. One root logger; modules call `log = structlog.get_logger(__name__)`. Every log line includes `sweep_id` / `cell_id` / `trial_id` from a `contextvars`-bound context. No `print` calls in `src/`. Log level via `LCRC_LOG_LEVEL` env var (defaults to INFO).

**Error model.** Custom exception hierarchy in `src/lcrc/errors.py`:

```python
class LcrcError(Exception): ...                 # base, never raised directly
class ConfigError(LcrcError): ...               # invalid profile, bad TOML, missing axis
class CatalogError(LcrcError): ...
class BackendError(LcrcError): ...              # base for backend issues
class BackendStartupError(BackendError): ...
class BackendCrashError(BackendError): ...
class HarnessError(LcrcError): ...
class HarnessTimeoutError(HarnessError): ...
class HarnessCrashError(HarnessError): ...
class TaskError(LcrcError): ...
class StorageError(LcrcError): ...
class BudgetExceededError(LcrcError): ...
```

The orchestrator catches `BackendError`/`HarnessError`/`TaskError` per cell or per trial — they mark the unit as failed and continue. `BudgetExceededError` and `StorageError` abort the sweep cleanly.

**Configuration loading.** Layered, in priority order: CLI flags > env vars (`LCRC_*`) > project `.lcrc/config.toml` (walks up from CWD) > user `~/.lcrc/config.toml` > shipped defaults. Implemented as a single `pydantic-settings` model. XDG-aware (`$XDG_DATA_HOME` / `$XDG_CONFIG_HOME` overrides if set).

**Concurrency.** The orchestrator is **synchronous** at the cell level (one cell at a time). The metrics collector samples in a background thread. The HTTP server (FastAPI) is async. Bridge: orchestrator emits events through an `asyncio.Queue` exposed to the FastAPI app via dependency injection. Subprocess management uses `subprocess.Popen` (sync) inside the orchestrator and `asyncio.create_subprocess_exec` only inside the FastAPI request lifecycle.

**Observability.** Per-cell timing breakdown is structured log fields, queryable later from the events table. Optional Prometheus metrics endpoint behind a `--metrics` flag (v1.1; not v1).

**Dependency injection.** Constructor injection only. Each module exposes a typed factory: `def build_orchestrator(db: Db, catalog: CatalogService, ...) -> Orchestrator`. No globals, no service locators. FastAPI uses its own DI for routes.

**Versioning.** `lcrc.__version__` from `importlib.metadata`. Every sweep records the lcrc version, every backend/harness records its detected build version, every catalog entry has a sha256 of the model file when downloaded. The schema_version table guards SQLite migrations.

**Retry and idempotency.** The orchestrator never auto-retries within a sweep (one shot per trial; reproducibility > resilience). On `lcrc run --resume`, only cells with `status='pending'` or `status='running'` (likely interrupted) are picked up — `running` rows are reset to `pending` first.

## v1 scope (concrete)

**Backends** (lifecycle-managed): `llama.cpp` via `llama-server`, `Ollama` via `llama-swap`. Both spawned/torn down per cell with the requested params; build version recorded.

**Harnesses**: `OpenCode` (shell out to `opencode run`); built-in `MiniBashHarness` (single-tool bash agent in the spirit of mini-swe-agent — internal Python implementation, no external dep).

**Tasks (5 total in v1)**:
- `CapabilityTask`: HumanEval (coding), MMLU-Pro slice (reasoning), SWE-bench-Verified-Lite or GAIA-mini (agentic). Sourced from the [`inspect_evals`](https://github.com/UKGovernmentBEIS/inspect_evals) catalog.
- `PerfTask`: single-stream throughput, concurrent-burst (workload patterns ported from Bench360 ideas — implemented internally to avoid dragging in their heavier deps).

**Sweeps**: single profile-based config concept (see "The sweep concept" section above). Canonical command is `lcrc run --profile <name|path>`. Shipped profiles: `discover` (full autonomous sweep), `weekly` (template), `smoke` (CI fixture). `lcrc discover` is a convenience alias.

**Reports**: `lcrc report --out report.md|.html|.json` — Pareto front per task + baked profiles (`coding-fast`, `long-context-quality`, `low-resource`).

**CLI surface**: `lcrc {discover, run, report, models, profile, serve, version}`.

**Frontend** (`lcrc serve`): read-only dashboard (browse runs, pivot tables, drill-down with transcripts + metrics), run launcher (pick profile or build sweep config, click Run), live progress (currently-running cell, queue, ETA, live tok/s, kill button).

**Cost guardrails**: per-cell timeout (default 5 min, configurable), per-run wall-clock budget (default 8h), pre-flight memory check, graceful SIGINT, resume from last-completed cell.

### Milestone details

Each entry below is the implementation brief for one milestone: **rationale** (why this exists, what it unlocks), **public interface** (signatures other milestones depend on), **internal notes** (key implementation decisions), **edge cases** (things to handle / not break), **test plan** (unit + integration cases the agent must implement).

#### M0 — Project foundation

- **Rationale:** A clean, strict, reproducible Python skeleton so every subsequent milestone inherits the same quality bar without arguing about config.
- **Public interface:** none. Establishes `pyproject.toml` (uv-managed), `pyrightconfig.json` (strict), `ruff.toml` (selecting `E,F,I,UP,B,SIM,RUF,ANN,N,PLR,PTH,TID`), `.pre-commit-config.yaml` (ruff + pyright + conventional commits), `tests/conftest.py` (shared fixtures stub), `.github/workflows/ci.yml` (lint + type + unit on macOS-latest + ubuntu-latest).
- **Internal notes:** `pyright` strict; ban `Any` in `src/` (`reportAny=true`). `pydantic` v2, `pydantic-settings`. Coverage target ≥ 80% configured in `pyproject.toml`. `uv` is the only env manager.
- **Edge cases:** lockfile committed; Python `>=3.12` required (TypedDict NotRequired, native Self, PEP 695); pre-commit must run on the same Python as CI.
- **Test plan:** CI must pass with empty src; pre-commit must reject a deliberately bad commit; `uv sync && uv run pyright && uv run ruff check && uv run pytest` returns 0 on a fresh checkout.

#### M1 — SQLite storage layer

- **Rationale:** The single source of truth. Every other module reads/writes through this typed repo. Without it nothing else has anywhere to land.
- **Public interface:**
  ```python
  class Db:
      def __init__(self, path: Path) -> None: ...
      def migrate(self) -> None: ...                                  # apply pending migrations
      def upsert_machine(self, fp: HardwareFingerprint) -> MachineId: ...
      def insert_sweep(self, s: SweepRow) -> SweepId: ...
      def insert_cells(self, cells: list[CellRow]) -> None: ...
      def update_cell_status(self, cell_id: CellId, status: str, **fields: Any) -> None: ...
      def insert_trial(self, t: TrialRow) -> TrialId: ...
      def insert_task_result(self, r: TaskResultRow) -> None: ...
      def insert_metric_samples(self, samples: Iterable[MetricSampleRow]) -> None: ...   # batched
      def insert_metric_aggregate(self, a: MetricAggregateRow) -> None: ...
      def list_sweeps(self, machine_id: MachineId | None = None) -> list[SweepSummary]: ...
      def get_sweep(self, sweep_id: SweepId) -> SweepDetail: ...
      def get_cells(self, sweep_id: SweepId, *, status: str | None = None) -> list[CellRow]: ...
      def get_trials(self, cell_id: CellId) -> list[TrialRow]: ...
      def get_task_results_for_sweep(self, sweep_id: SweepId) -> dict[TaskId, list[TrialResult]]: ...
      def append_event(self, e: ProgressEvent) -> None: ...
      def get_events_since(self, sweep_id: SweepId, after_event_id: int) -> list[ProgressEvent]: ...

  # src/lcrc/storage/fingerprint.py
  def detect_fingerprint() -> HardwareFingerprint: ...                 # platform-dispatched
  ```
- **Internal notes:** Migrations live in `src/lcrc/storage/migrations/000N_*.sql`. `Db` opens one connection per call (SQLite is fine; WAL handles concurrency). Foreign keys ON. JSON blobs serialized via `pydantic.model_dump_json()`. No ORM — plain SQL with typed result rows (`pydantic` `from_attributes=True`).
- **Edge cases:** schema migration ordering (must be lexicographic); concurrent reads during writes (WAL); large `metric_samples` inserts (use `executemany`); fingerprint stability across restarts (hash a stable subset, not e.g. uptime).
- **Test plan:**
  - Unit: every public method round-trips a row. `detect_fingerprint()` returns identical id across two calls. Migration test: empty DB → schema_version=1 with all tables created.
  - Property: insert N=10 000 metric samples in one transaction completes < 500 ms.
  - Integration: corrupt a partial write by killing mid-transaction (simulated); reopen, no rows partially visible.

#### M2 — Metrics collector (macOS Apple Silicon)

- **Rationale:** Power and unified-memory telemetry on Apple Silicon is the headline reason a user would even want lcrc on a Mac ("will this melt my MacBook?"). This is what makes the recommendation actionable.
- **Public interface:** the `MetricsCollector` Protocol from the design section, plus a concrete `MacosCollector(MetricsCollector)`.
- **Internal notes:** CPU/RAM via `psutil`. GPU% / power via `powermetrics` (subprocess, sampled at 1 Hz, JSON output via `--format plist` parsed). Sudo cached once per session; a one-time prompt is acceptable, then `~/.lcrc/cache/powermetrics_auth` records the grant timestamp. ANE usage parsed where present. Background thread + `queue.Queue` for samples; main thread joins on `stop()`.
- **Edge cases:** `powermetrics` requires sudo — degrade gracefully: `power_w=None` instead of crashing. macOS Sequoia changed some `powermetrics` keys; pin to a parser that tolerates missing keys. Process under measurement may exit before stop — handle `psutil.NoSuchProcess`.
- **Test plan:**
  - Unit: `sample_window(pid=os.getpid(), duration=2)` returns ≥ 1 sample with non-zero RAM. Mock `powermetrics` subprocess to test parser. Test the no-sudo fallback.
  - Integration (mac CI runner): real 5s sample of a sleep process, verify aggregate keys present.

#### M3 — Metrics collector (Linux NVIDIA)

- **Rationale:** Cover the second tier-1 platform with the same Protocol so the orchestrator stays platform-agnostic.
- **Public interface:** `LinuxNvidiaCollector(MetricsCollector)`.
- **Internal notes:** `pynvml` for GPU util, VRAM, power, temp. `psutil` for CPU/RAM. Init NVML once per process, finalize on collector destruction. Multi-GPU: report sum of VRAM, max of power, sum of util percentages clamped per-GPU.
- **Edge cases:** Non-NVIDIA host → `supported() == False`. Driver version mismatch (`pynvml` raises `NVMLError_DriverNotLoaded`). Containerized environments with no `/dev/nvidia*` exposed — same as no-GPU.
- **Test plan:**
  - Unit: stub `pynvml` calls and verify aggregation logic.
  - Integration (gated by `LCRC_TEST_GPU=1`): real sample on actual GPU.

#### M4 — Backend protocol + llama.cpp driver

- **Rationale:** The first usable backend; also the reference implementation that every other backend driver mimics. Without M4 the orchestrator has nothing to spawn.
- **Public interface:** the `Backend` Protocol from design section, plus `LlamaCppBackend(Backend)`.
- **Internal notes:** Spawns `llama-server --model <gguf> --ctx-size <n> --cache-type-k <kv_quant> --cache-type-v <kv_quant> --n-gpu-layers <n> --port <auto> --host 127.0.0.1 --no-webui`. Picks a free port with `socket`. Polls `/health` with exponential backoff up to 30 s. Build version captured from `--version`. Preflight: POST a 1-token generation, assert non-empty response, capture echoed model id and ctx. Teardown: `SIGTERM` then `SIGKILL` after 5 s.
- **Edge cases:** binary not on PATH → `BackendStartupError("llama-server not found")`. Port collision (rare with auto-pick but possible). Model file missing → fail fast in preflight, not in `prepare()`. ctx_len > what model supports → llama-server clips it; capture the clip in `model_endpoint.max_context`.
- **Test plan:**
  - Unit: arg construction from `BackendSpec` is correct (parametrized).
  - Integration: spawn against a tiny GGUF fixture (TinyLlama-1.1B Q4_K_M ~ 600 MB committed under `tests/fixtures/`); preflight → 1-token round-trip → teardown leaves no zombie process. Test SIGKILL fallback by simulating a hung process.

#### M5 — Backend driver: Ollama via llama-swap

- **Rationale:** Cover the most popular local backend without inheriting Ollama's silent-unload bug.
- **Public interface:** `OllamaBackend(Backend)` (same Protocol as M4).
- **Internal notes:** Drives [`llama-swap`](https://github.com/mostlygeek/llama-swap) — a process supervisor that starts a dedicated llama.cpp server per model with fixed config (no implicit unloading). Generates `llama-swap` config on the fly per cell. Endpoint speaks OpenAI-compatible (llama-swap's frontend). For a pure Ollama option (no llama-swap), use `/api/generate` and validate the `context` field in the response matches request — but mark this as "v1.1 experimental" since it inherits the silent-unload risk.
- **Edge cases:** `llama-swap` not installed → `BackendStartupError` with a hint to install. Model not pulled to Ollama yet → optionally call `ollama pull` (gated by `--auto-pull` flag) or fail with actionable message. Verify `model_endpoint.max_context` matches request — Ollama silently clips otherwise.
- **Test plan:**

#### M6 — Catalog: curated baseline + hardware-fit gate

- **Rationale:** Without a curated catalog there's nothing for `from_catalog` to expand into. The hardware-fit gate is the safety rail that prevents OOMs in autonomous mode.
- **Public interface:**
  ```python
  class CatalogService:
      def __init__(self, curated_path: Path, user_path: Path | None = None) -> None: ...
      def list(self, *, fit_for: HardwareFingerprint | None = None,
               family_whitelist: list[str] | None = None,
               max_size_b: float | None = None,
               backend: BackendName | None = None) -> list[CatalogEntry]: ...
      def get(self, entry_id: CatalogEntryId) -> CatalogEntry: ...
      def add_user_entry(self, entry: CatalogEntry) -> None: ...

  # src/lcrc/catalog/fit.py
  def fits(entry: CatalogEntry, quant: QuantTag, hardware: HardwareFingerprint,
           safety_factor: float = 0.85) -> tuple[bool, str]: ...   # (fits, reason)
  ```
- **Internal notes:** `curated.yaml` ships ~15 entries (Llama-3.x, Qwen3 family, Gemma3, Phi4, Mistral, DeepSeek-Coder, etc.). Each has `supported_quants`, `supported_backends`, and one `ModelSource` per (backend, quant). `fits()` estimates working-set bytes ≈ `parameter_count * bytes_per_weight(quant) * 1.15` (overhead) + KV cache estimate; compares to available memory (VRAM for discrete GPU, RAM for unified-memory).
- **Edge cases:** YAML schema drift (validate on load). Apple Silicon unified memory — VRAM == RAM. Catalog entry with no source for the requested backend → returned by `list()` without that backend; orchestrator filters.
- **Test plan:**
  - Unit: load curated.yaml; expect ≥ 10 entries, schema valid. `fits()` parametrized: 7B model q4_k_m fits 8GB unified memory, 70B q4 doesn't fit 16GB.
  - Unit: `list(fit_for=...)` returns expected subset against simulated hardware specs.

#### M8 — Task protocol + capability tasks (Inspect wrapper)

- **Rationale:** Capability tasks are how we measure quality. Wrapping Inspect AI lets us inherit 200+ pre-built evals, model-as-judge graders, and a transcript format that other tools already understand.
- **Public interface:**
  ```python
  class CapabilityTaskBase(CapabilityTask):
      """Shared base for Inspect-derived tasks."""
      id: TaskId
      kind: Literal["capability"] = "capability"

      @abstractmethod
      def build_inspect_task(self, n_problems: int | None) -> "inspect_ai.Task": ...

      def evaluate(self, harness: Harness, endpoint: ModelEndpoint,
                   workdir: Path, n_problems: int | None = None) -> CapabilityScore:
          inspect_task = self.build_inspect_task(n_problems)
          # bridge: harness drives Inspect's solver via a custom solver wrapper
          ...

  class HumanEvalMini(CapabilityTaskBase):
      id = TaskId("humaneval_mini")
      def build_inspect_task(self, n_problems): return humaneval(limit=n_problems or 5)
  ```
- **Internal notes:** The bridge is the tricky part — Inspect normally drives the model directly; here we want the **harness** to drive the model so we can measure harness × model interaction. Solution: implement an Inspect `Solver` that delegates a single problem to `harness.run(endpoint, single_problem_task, workdir)`. Inspect still grades the output, computes the score, writes the `.eval` file.
- **Edge cases:** Inspect dataset download failures (cache aggressively under `~/.lcrc/cache/datasets/`). HumanEval requires Docker for code execution graders — guard with a `--unsafe-codegen` flag for users without sandboxing. Per-problem timeouts (Inspect handles internally).
- **Test plan:**
  - Unit: solver wrapper passes harness output through correctly with mocked harness.
  - Integration: full HumanEval-mini (5 problems) against a stubbed harness that returns canned correct answers → score 1.0; against a stub returning empty strings → score 0.0.

#### M9 — Capability task suite + perf tasks

- **Rationale:** Three capability tasks span the quality axes the user cares about (coding, reasoning, agentic). Two perf workloads cover the system-metrics axis.
- **Public interface:** concrete classes `MmluProSlice`, `SwebenchVerifiedLite` (or `GaiaMini`), `SingleStreamPerf`, `ConcurrentBurstPerf`. Same Protocol as M8.
- **Internal notes:** `SingleStreamPerf` issues N sequential requests (default 50) of fixed prompt+gen size, measures TTFT, ITL percentiles, TPS. `ConcurrentBurstPerf` issues K parallel requests (default 8), measures aggregate throughput. Neither uses a harness — they call the endpoint directly via `httpx`.
- **Edge cases:** SWE-bench requires Git + Docker; auto-skip with a warning if absent. Perf workloads must respect `endpoint.max_context` to avoid skewing results when ctx is clipped.
- **Test plan:** each task end-to-end against a stub endpoint returning fixed-rate tokens; assert PerfMetrics fields are computed correctly.

#### M10 — Harness protocol + mini_bash harness

- **Rationale:** A first-party minimal harness is the experimental control — when comparing harnesses, you need a baseline that isolates the model.
- **Public interface:** `MiniBashHarness(Harness)`.
- **Internal notes:** Single-tool agent in the spirit of mini-swe-agent. System prompt: "You have one tool: `bash(cmd)`. Solve the task." Loop: model emits a JSON tool call → execute in a sandboxed cwd → append observation → continue. Uses `endpoint` directly via OpenAI-compatible chat completions. Transcript: one JSONL line per turn with `{role, content, tool_call?, observation?}`. Stop conditions: model emits `final_answer`, tool budget exhausted (default 30 turns), wall-clock timeout.
- **Edge cases:** Model emits malformed tool calls — count as a turn, return parse-error observation, continue. Bash command times out (per-command 60s limit). Output truncation at 4 KB per observation.
- **Test plan:**
  - Unit: tool-call parser handles malformed JSON gracefully.
  - Integration: against tiny model + llama.cpp, solve "create a file named hello.txt with content world". Assert transcript shows the bash call and `final_answer`.

#### M11 — Harness driver: OpenCode subprocess

- **Public interface:** `OpencodeHarness(Harness)`.
- **Internal notes:** Run `opencode run --provider openai --base-url <endpoint.base_url> --model <endpoint.model_name> --no-color "<task_prompt>"` in a temp workdir. Parse transcript from OpenCode's JSON output mode (or scrape stdout if needed). Capture exit code; non-zero → `HarnessCrashError`.

#### M12 — Orchestrator core + profile loader

- **Rationale:** The heart of the framework. Reads a profile, expands it into cells, executes them with safety rails, persists everything. After M12 lcrc is functionally a benchmark runner (just without tiered dropout, the report, the CLI polish, or the UI).
- **Public interface:**
  ```python
  class Profile(BaseModel):                 # full schema from the Profile TOML section
      meta: ProfileMeta
      execution: ExecutionConfig
      axes: ProfileAxes
      tasks: TaskSelection

  def load_profile(name_or_path: str) -> Profile: ...        # resolves name → path → TOML → Profile

  class Orchestrator:
      def __init__(self, db: Db, catalog: CatalogService, registry: PluginRegistry,
                   metrics: MetricsCollector, event_bus: EventBus, root_dir: Path) -> None: ...
      def start(self, profile: Profile, *, resume: bool = False) -> SweepId: ...
      def cancel(self, sweep_id: SweepId) -> None: ...

  class PluginRegistry:
      backends: dict[BackendName, Backend]
      harnesses: dict[HarnessName, Harness]
      tasks: dict[TaskId, Task]

  class EventBus:
      def publish(self, event: ProgressEvent) -> None: ...   # writes to db.events + asyncio.Queue subscribers
      def subscribe(self, sweep_id: SweepId, after_event_id: int = 0) -> AsyncIterator[ProgressEvent]: ...
  ```
- **Internal notes:** Cell expansion: cartesian product over resolved axis values, deduped by `frozenset` of axis values. Workdir per trial: `~/.lcrc/runs/<sweep_id>/cells/<cell_id>/<trial_idx>/`. SIGINT installed via `signal.signal(SIGINT, handler)` only while `start()` is on the call stack; handler sets a `cancelled` flag checked at every cell boundary. Resume: query cells with `status IN ('pending','running')`, reset 'running' to 'pending', continue.
- **Edge cases:** Empty cell expansion (e.g., all `pinned` and one axis has no values) → `ConfigError`. Backend startup failure on every cell of a model → mark all cells of that model preflight_failed quickly. Disk-full while writing transcripts → abort sweep, mark `sweep.status='failed'`.
- **Test plan:**
  - Unit: cell expansion produces expected combinations for every TOML example in the design doc.
  - Unit: profile schema validation rejects invalid TOMLs (per-field).
  - Integration: 2-cell `dropout='off'` profile runs end-to-end with stub backend/harness/task; SQLite contains expected rows. Mid-run SIGINT leaves DB in consistent state.

#### M13 — Orchestrator: tiered dropout policy

- **Rationale:** The autonomous-iteration secret sauce. Without it, `discover` would run all 21 600 cells. With it, only the survivors of each stage advance.
- **Public interface:**
  ```python
  class TieredDropoutPolicy:
      def stage1_cells(self, all_cells: list[CellSpec]) -> list[CellSpec]: ...   # all, smoke task only, N=1
      def promote_to_stage2(self, stage1_results: list[TrialResult],
                            cfg: TieredConfig) -> tuple[list[CellId], dict[CellId, str]]: ...
      def stage2_cells(self, promoted: list[CellId], full_axes: ProfileAxes) -> list[CellSpec]: ...
      def promote_to_stage3(self, stage2_results: list[TrialResult],
                            cfg: TieredConfig) -> tuple[list[CellId], dict[CellId, str]]: ...
  ```
- **Internal notes:** Stage 1 = every cell × smoke capability task (5 problems) × N=1 trial. Drop: crashed, score < `stage1_quality_floor`. Stage 2 = survivors × full quant/context expansion (per-model defaults from catalog) × N=1 × shorter capability+perf suite. Promote: top `stage2_pareto_keep` fraction on Pareto front of (quality, tps, peak_mem). Stage 3 = top-K × full task suite × N=3.
- **Edge cases:** All cells fail Stage 1 → sweep completes with empty recommendation, status='completed' but report says "no viable combinations." Stage 2 produces fewer cells than `stage3_top_k` → promote all. User sets unrealistic `stage1_quality_floor` (1.0) → handle and warn.
- **Test plan:**
  - Unit: dropout policy on synthetic stage1 results (some pass, some fail, some crash) produces expected promotion + reasons.
  - Integration: shipped `discover.toml` against a 4-cell mini-catalog runs all 3 stages with stage_decisions table populated.

  ```python
  def list_profiles(profiles_dir: Path) -> list[ProfileSummary]: ...
  def show_profile(name: str, profiles_dir: Path) -> Profile: ...
  ```

- **Edge cases:** `init` requested before any `discover` ran → suggest running `lcrc discover` first. Profile name collision → error unless `--force`.

#### M15 — Reporter: Pareto + baked profiles

- **Rationale:** The headline output. Without M15 the framework is just a data collector — M15 turns numbers into recommendations.
- **Public interface:**
  ```python
  class ParetoRecommender(Recommender):
      def recommend(self, sweep_id: SweepId,
                    per_task_results: dict[TaskId, list[TrialResult]],
                    weights: ProfileWeights | None = None) -> list[Recommendation]: ...

  def render_markdown(recs: list[Recommendation], sweep: SweepDetail) -> str: ...
  def render_html(recs: list[Recommendation], sweep: SweepDetail) -> str: ...
  def render_json(recs: list[Recommendation], sweep: SweepDetail) -> str: ...
  ```
- **Internal notes:** Per task, normalize quality scores to [0,1]. Compute the Pareto front over (quality desc, tps desc, peak_memory asc, energy_per_token asc). For each baked profile, project to a single axis ordering and pick the front member ranked first. "Balanced" = closest-to-utopia point in the normalized space. Each `Recommendation` includes `rationale` text built from the dominating relations ("dominates X by 12% TPS at equal quality").
- **Edge cases:** Empty results → return empty list, render document still has skeleton structure. Single cell → it wins every profile trivially (mark with note). Tied quality + tied tps → secondary sort by memory.
- **Test plan:**
  - Unit: Pareto computation against synthetic 2-D and 4-D fixtures with known answers.
  - Golden-output: render_md against a fixture sweep produces a stable markdown file (snapshot).

#### M18 — WebSocket live progress

- **Rationale:** Live progress is the UX moment that justifies the frontend (otherwise tail -f a log).
- **Public interface:** `GET /api/ws/sweeps/{sweep_id}/progress?after_event_id=N`. Server streams `ProgressEvent` JSON messages.

We do **not** vendor or wrap Bench360, mini-swe-agent, or other custom repos directly — only borrow conceptual patterns. ## Quality bar

- `pyright` strict mode passes (no `Any` leaks across module boundaries)
- `ruff check` + `ruff format` clean
- `pytest` ≥80% coverage on `src/lcrc/{orchestrator, backends, harnesses, tasks, storage, reporting}`
- Integration test boots a tiny GGUF model end-to-end via llama.cpp on a CI runner
- All public functions have type-annotated signatures; data crossing boundaries is `pydantic` BaseModel
- Pre-commit hooks: ruff, pyright, conventional-commit lint

## Verification (end-to-end)

1. **Install & sanity**: `uv sync`, `uv run lcrc --version` returns the version. `uv run lcrc models list` shows the curated catalog.
2. **Smoke run** (tiny model): `uv run lcrc run --profile tests/fixtures/smoke.toml` with one model (TinyLlama Q4_K_M), one backend (llama.cpp), one harness (`mini_bash`), one task (HumanEval-mini = 5 problems). Confirms full happy path: spawn → preflight → harness loop → grade → metrics → SQLite write.
3. **Discover**: `uv run lcrc discover` (or `uv run lcrc run --profile discover`) runs the tiered sweep across the curated catalog filtered by the hardware-fit gate. Verify Stage 1 → Stage 2 → Stage 3 transitions are logged with dropout reasons in the `stage_decisions` table.
4. **Profile init flow**: `uv run lcrc profile init weekly --from-sweep <sweep_id>` generates a `weekly.toml` with backend + harness pinned from the discover run's recommendations. `uv run lcrc run --profile weekly` then sweeps only the unpinned axes (model/quant/context).
5. **Report**: `uv run lcrc report <sweep_id> --out /tmp/r.md` produces a Pareto-front markdown with `coding-fast`, `long-context-quality`, `low-resource` sections. Same data via `--format json` is parseable; `--format html` renders standalone.
6. **Frontend**: `uv run lcrc serve` boots FastAPI on `localhost:8080`, the bundled SPA loads, the previous run is browsable, the run launcher fires a new sweep, live progress streams over WebSocket, kill button stops it cleanly.
7. **Resume**: `Ctrl-C` mid-sweep, then `uv run lcrc run --profile <name> --resume` picks up at the last incomplete cell with no double-trial in SQLite.
8. **Cross-machine**: copy `runs.db` from a second machine into `~/.lcrc/runs.db` (or use `--db <path>`); reports filter/compare by `machine_id`.
9. **Reconnect**: open the live-progress page mid-run, hard-refresh; SPA reconnects via WebSocket using `after_event_id` cursor and replays missed events without gaps.

## Explicitly out of scope for v1 (designed-for, not implemented)

- Backend plugins: vLLM, LM Studio, SGLang
- Harness plugins: Claude Code (needs LiteLLM proxy), pi.dev
- Linux AMD ROCm telemetry, Windows polish
- User-weighted recommendation sliders (v1 ships fixed Pareto + baked profiles)
- Optuna/NSGA-II alternative sweep strategy
- Distributed/multi-machine coordination at runtime (cross-machine *comparison* in the report works via shared `runs.db`)
- In-UI model installation, in-UI task editing
- Power telemetry without sudo on macOS (we'll ask the user once and cache the auth)

Each deferred item slots into an existing typed interface (`Backend`, `Harness`, `Task`, `MetricsCollector`, `Recommender`).
